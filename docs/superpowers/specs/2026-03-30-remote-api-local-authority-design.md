# Remote API 本地主导同步设计

日期：2026-03-30

## 背景

当前项目已经有 `DatabaseManager`、本地 SQLite、任务缓存、周期同步和基础的远端 API 调用能力，但行为仍然偏向“本地数据库 + 远端增量同步”。`remote-api.md` 明确要求：本地访问远端生产数据时，应优先调用 HTTP API，而不是直接读远端 SQLite 文件。

用户新增约束：

1. PC 端以本地为准。
2. 远端与本地发生冲突时，需要逐条选择保留本地还是接受远端。
3. 尽量复用现有本地代码，除非继续复用会让结构明显臃肿。

## 目标

在不引入臃肿新层的前提下，让现有桌面端通过远端 HTTP API 完成连接、拉取、回写和冲突处理，同时保持本地 SQLite/内存缓存仍然是 PC 端的权威副本。

## 非目标

- 不引入单独的 Repository / Service 大重构。
- 不改造为“远端绝对主库、本地纯缓存”的架构。
- 不通过共享网络盘、SMB、挂载目录等方式直接读取远端 SQLite。
- 不一次性重写全部同步逻辑；优先沿用现有 `DatabaseManager`、冲突弹窗和缓存机制。

## 方案比较

### 方案 A：远端优先，本地仅缓存

做法：启动后先拉远端，读写都尽量直接打 API，本地只做离线兜底。

优点：
- 行为简单。
- 与 `remote-api.md` 的“优先 API”字面一致。

缺点：
- 违背“PC 端以本地为准”。
- 一旦远端返回更新，容易在启动或轮询时直接覆盖本地。
- 现有本地缓存、冲突确认和离线写入价值会被削弱。

### 方案 B：本地主导，远端作为同步来源与同步目标（推荐）

做法：保留本地 SQLite + 内存缓存为主，远端通过 API 进行拉取与回写。拉取时只把“无本地对应项”自动补入本地；对已有本地数据的远端变更，生成逐条冲突项，由用户选择接受远端或保留本地。

优点：
- 完全符合用户要求。
- 最大化复用现有 `DatabaseManager`、冲突弹窗、缓存和同步线程。
- 离线能力保留，风险可控。

缺点：
- 需要把当前冲突逻辑从“普通任务”扩到“通用远端变更项”。
- 需要补充启动同步时机和认证补偿逻辑。

### 方案 C：独立同步引擎 + 领域适配层

做法：新增 RemoteSyncEngine、TaskSyncAdapter、ScheduledTaskSyncAdapter 等，把本地存储与远端 API 完全解耦。

优点：
- 长期结构最清晰。
- 更容易扩展多种远端源。

缺点：
- 超出当前需求。
- 改动面大，回归风险高。
- 与“尽量复用本地代码”不符。

## 采用方案

采用方案 B：本地主导、远端 API 同步、逐条冲突决策。

## 设计概览

### 1. 远端连接与认证

远端配置继续从 `remote_config.json` 读取，但字段扩展为：

```json
{
  "api_base_url": "https://example.com",
  "api_token": "token",
  "username": "alice"
}
```

认证策略：

- `GET /api/health` 与 `POST /api/users` 走匿名请求。
- 其他业务接口带 `Authorization: Bearer <token>`。
- 若业务请求返回 `401`，且本地已配置 `username + api_token`，客户端自动调用 `POST /api/users` 注册一次，然后重试原请求一次。
- 自动注册只尝试一次，避免死循环。

### 2. 启动同步时机

不能在 `DatabaseManager.__init__()` 中直接把远端数据套用到本地，因为那时 UI 侧冲突监听器尚未挂上，会破坏“逐条选择”的要求。

因此启动流程改为：

1. `DatabaseManager` 只完成本地数据库和缓存初始化。
2. `QuadrantWidget` 注册冲突监听器。
3. UI 就绪后显式调用 `bootstrap_remote_sync()`。
4. `bootstrap_remote_sync()` 只负责：
   - 健康检查。
   - 拉取远端普通任务。
   - 拉取远端定时任务。
   - 生成远端变更项或补入无冲突数据。

### 3. 冲突模型

把现有“普通任务远端修改待确认”扩展为统一的远端变更模型：

```python
{
  "change_key": "task:abc" 或 "scheduled:xyz",
  "entity_type": "task" | "scheduled_task",
  "entity_id": "abc",
  "change_type": "create" | "update" | "delete",
  "title": "展示标题",
  "local_record": {...} | None,
  "remote_record": {...} | None,
}
```

规则：

- 远端新增且本地不存在：可以直接补入本地，或按 UI 统一弹窗展示为“新增项待确认”。
- 远端更新且本地存在：视为冲突，必须逐条决策。
- 远端删除且本地仍存在：视为冲突，必须逐条决策。
- 用户选择“接受远端”：本地覆盖为远端版本，必要时删除本地记录。
- 用户选择“保留本地”：本地保持不变，并在下一次上传时把本地版本回写远端。

## 普通任务设计

### 本地写入

保持现有模式：

- UI 编辑后写内存缓存。
- 标记 `sync_status = modified`。
- 定时同步/退出同步/手动同步再上传到远端。

理由：

- 普通任务位置变化频繁，不适合每次拖动都实时打 API。
- 现有缓存与 flush 机制已经适合这一类高频编辑。

### 远端拉取

`sync_from_server()` 行为调整为：

- 远端数据与本地无冲突时才自动落地。
- 对已有本地记录的远端更新/删除，不直接覆盖，转为待确认项。
- UI 弹窗逐条展示，用户勾选接受远端；未勾选项保持本地并在后续上传回远端。

### 历史记录

`get_task_history(task_id)` 优先调用 `/api/tasks/<task_id>/history`；远端失败时回退到本地 `task_history` 表。

## 定时任务设计

### 本地写入

定时任务变更频率低于普通任务，因此采用“本地写入成功后立即 best-effort 回写远端”的策略：

- `create_scheduled_task()`：本地 `INSERT OR REPLACE` 后立即 `POST /api/scheduled_tasks`。
- `update_scheduled_task()`：本地更新后立即 `POST /api/scheduled_tasks`。
- `delete_scheduled_task()`：本地删除后立即 `DELETE /api/scheduled_tasks/<id>`。

若远端失败：

- 不回滚本地成功结果。
- 记录日志。
- 仍允许周期同步再次兜底回写。

### 远端拉取

定时任务也应遵守“本地主导”：

- 本地不存在的远端定时任务，可补入本地。
- 本地已存在且远端更新时间更新时，不自动覆盖本地。
- 该类冲突也应进入统一的远端变更确认流程，而不是简单按时间戳覆盖。

## UI 设计

复用现有远端冲突确认弹窗，但展示项改为通用远端变更摘要：

- `[新增] 任务标题`
- `[修改] 任务标题 - 差异`
- `[删除] 任务标题`
- 对定时任务可在标题前补实体前缀，例如 `[定时任务][修改] 每周复盘`。

交互规则：

- 勾选：接受远端。
- 不勾选：保留本地。
- 确认后批量应用。
- 若存在“保留本地”的项，需要立即或在下一轮同步中把本地内容回写远端。

## 文件级职责

### `database/database_manager.py`

职责：

- 远端请求封装。
- 自动注册补偿。
- 本地缓存与 SQLite 维护。
- 普通任务/定时任务的同步与冲突构造。
- 冲突应用与回写。

要求：

- 保持为主协调点。
- 允许适度抽小 helper，但不拆出一堆新模块。

### `core/quadrant_widget.py`

职责：

- 在监听器挂载后触发启动同步。
- 展示远端冲突确认弹窗。
- 在用户确认后刷新界面。

### `config/remote_config.py`

职责：

- 支持 `username` 的保存、读取和测试连接。

### `tests/test_database_manager_remote.py`

职责：

- 覆盖远端健康检查/注册重试/启动同步/历史优先远端/定时任务本地优先写入等行为。

## 错误处理

- 健康检查失败：不阻塞本地启动，只记录日志。
- 自动注册失败：不再重复尝试，当前请求按失败处理。
- 远端拉取失败：保留本地数据，不清空缓存。
- 冲突确认失败：不改变本地现状。
- 定时任务远端回写失败：不回滚本地已成功修改。

## 测试策略

### 单元测试

至少覆盖：

1. `DatabaseManager` 初始化不自动执行远端套用。
2. `bootstrap_remote_sync()` 在显式调用后才进行健康检查与拉取。
3. 业务请求 `401` 时自动注册并重试一次。
4. `get_task_history()` 优先远端、失败回退本地。
5. `create/update/delete_scheduled_task()` 本地成功后即使远端失败也保留本地。
6. 普通任务与定时任务的远端冲突均进入待确认集合，而不是直接覆盖。

### 手工验证

1. 启动应用，无远端配置时功能不变。
2. 配置远端后启动，先显示本地数据，再在 UI 就绪后拉取远端。
3. 构造普通任务冲突，验证逐条选择生效。
4. 构造定时任务冲突，验证不会静默覆盖本地。
5. 远端服务断开时，本地增删改仍可继续。

## 风险与缓解

### 风险 1：启动同步过早

缓解：只允许 UI 在监听器注册后触发 `bootstrap_remote_sync()`。

### 风险 2：定时任务冲突逻辑与普通任务分叉

缓解：统一远端变更摘要结构与确认入口，减少两套判断分支。

### 风险 3：自动注册错误地掩盖鉴权问题

缓解：仅在首次 `401` 时尝试一次注册，注册后只重试一次原请求，并保留完整日志。

## 验收标准

满足以下条件视为完成：

1. 本地端不再依赖任何远端 SQLite 直连方式。
2. 远端业务数据访问通过 `remote-api.md` 描述的 HTTP API 完成。
3. PC 端仍以本地为准。
4. 普通任务冲突支持逐条选择本地或远端。
5. 定时任务冲突不会静默覆盖本地，且进入同样的决策流程或至少保守保留本地。
6. 启动、离线、退出等路径不破坏现有本地使用体验。
