# 历史记录本地优先与同步设计

## 背景

当前任务详情中的历史弹窗与导出功能，会通过 `DatabaseManager.get_task_history()` 在开启远程配置时优先读取远端历史。

这会带来两个问题：

1. 远端历史不完整时，界面展示会比本地数据库实际历史更短。
2. UI 展示口径与最终本地落库状态可能不一致，容易让用户误判“数据库丢了历史”。

本次调整的目标，是让历史记录展示和最终同步策略都围绕“本地为主、远端参与合并”展开，并复用现有的任务冲突合并流程。

## 范围

本次只调整以下行为：

1. 历史弹窗读取逻辑。
2. 历史导出读取逻辑。
3. 任务冲突合并时的历史记录处理策略。
4. 本地与远端历史的落库/推送顺序。

本次不重做现有任务字段冲突判断与接受合并流程，不新增新的交互入口，不改动历史弹窗样式。

## 目标行为

### 1. UI 历史读取

历史弹窗与导出统一只读本地库。

`database/database_manager.py` 中的 `get_task_history()` 不再因为配置了远程接口而直接返回远端历史；它只返回当前本地数据库中的 `task_history` 内容。

结果是：

1. 历史弹窗始终展示本地最终状态。
2. 导出内容与弹窗展示保持一致。
3. UI 不再直接受远端历史是否完整影响。

### 2. 冲突合并时的历史策略

当本地任务与远端任务发生冲突时，继续沿用现有的“接受合并”流程，不单独新增新的历史合并交互。

在该既有流程内部，增加历史处理决策：

1. 判断是否把远端缺失于本地的历史追加到本地。
2. 判断是否把本地缺失于远端的历史追加到远端。

历史记录的合并是“追加”而不是“覆盖”；已有同一条历史时应去重，不应重复写入。

### 3. 落库顺序

当一次冲突合并被确认后，先把本地库写成最终一致状态，再进行远端推送。

顺序固定为：

1. 完成任务字段的本地最终合并。
2. 完成本地 `task_history` 的最终合并。
3. 以本地最终状态为基准，决定需要追加到远端的历史。
4. 再执行远端更新/推送。

这样可以保证：

1. UI 立即能看到最终结果。
2. 即使远端推送失败，本地仍保留完整且可信的历史。
3. 重试远端同步时有明确的本地真值来源。

## 数据规则

历史记录合并采用追加去重策略。

默认以 `(task_id, field_name, timestamp)` 作为历史记录唯一键，与当前表主键保持一致。对于本地与远端映射到同一唯一键的记录：

1. 保留本地已存在记录。
2. 不重复插入。

对于仅存在于一侧的记录：

1. 在接受合并后，允许追加到另一侧。

## 实现约束

1. `get_task_history()` 改为本地唯一读取源。
2. 历史弹窗与导出不直接访问远端接口。
3. 历史同步逻辑必须挂接在现有冲突合并链路中，而不是在 UI 查询时临时拼接远端与本地结果。
4. 历史推远端时只做追加去重，不做全量覆盖删除。

## 验收标准

1. 对于本地存在早于远端的历史任务，历史弹窗可显示完整本地历史。
2. 历史导出结果与历史弹窗一致。
3. 开启远程配置时，单独打开历史弹窗不会因为远端历史缺失而丢失本地历史。
4. 冲突合并后，本地库先成为最终结果，再进行远端推送。
5. 远端推送失败时，本地历史仍保持完整可见。
6. 已存在的历史记录不会因双向追加而重复插入。
